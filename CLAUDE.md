# CLAUDE.md

## 작업 범위 지정 커맨드

메시지 맨 앞에 아래 prefix를 붙여 작업 범위를 지정한다.

| prefix | 의미          | 문서 경로        | 작업 디렉토리 |
| ------ | ------------- | ---------------- | ------------- |
| `-b`   | backend 작업  | `backend/docs/`  | `backend/`    |
| `-f`   | frontend 작업 | `frontend/docs/` | `frontend/`   |

- prefix가 없으면 반드시 작업 범위를 사용자에게 되물을 것.
- 두 영역에 걸치는 작업이면 `-b -f` 또는 `-bf` 로 표기한다.

## 워크플로우 규칙

### 1. 문서 위치 규칙

- `backend` 관련 문서는 반드시 `backend/docs/` 폴더에 작성할 것.
- `frontend` 관련 문서는 반드시 `frontend/docs/` 폴더에 작성할 것.
- 문서 파일명에는 항상 `YYMMDD_` 형식의 날짜 prefix를 붙일 것. (시스템의 현재 날짜를 확인할 것)
  - 예: `260323_board-view-count-cookie-plan.md`

### 2. 코드 작성 이전 — 작업 계획서 필수

- 코드를 작성하기 전, 반드시 작업 계획서를 markdown 형식으로 상세히 작성할 것.

  계획서는 단순 작업 나열이 아니라, **문제 해결 및 의사결정 과정**을 포함해야 한다.

- 계획서에는 다음 내용을 포함할 것:
  
  - Problem (문제 정의)
    -  어떤 문제를 해결하려는가?
    - 이 문제가 왜 중요한가?
    - 해결하지 않을 경우 어떤 문제가 발생하는가?
  
  - Analyze (분석 및 선택지 검토)
    - 가능한 구현 방법/기술 스택은 무엇인가?
    - 각 선택지의 장단점은 무엇인가?
    - 최종적으로 어떤 방식을 선택했고, 그 이유는 무엇인가?
  
  - Action (구현 계획 및 설계)
    - 작업 목표 및 범위
    - 구현 접근 방식 및 설계 결정
    - 변경 대상 파일 목록
    - 주요 트레이드오프 (성능 vs 복잡도 등)
    - 예상되는 이슈 및 대응 방안
  
  - Result (검증 계획)
    - 어떤 방식으로 결과를 검증할 것인가?
    - 테스트 코드
      - 성능 측정 (ex. latency, throughput)
      - 로그 / 모니터링
      - 성공 기준은 무엇인가?
  

### 3. 코드 작성 시 — Context7 MCP 활용

- 코드 작성 시 **Context7 MCP**를 적극 활용하여 최신 라이브러리 문서를 참조할 것.
- 공식 문서 기반으로 작성하여 버전 불일치 및 오류를 사전에 방지할 것.

### 4. 테스트 — 관련 파일만 실행

- 코드 작성 완료 후 전체 테스트(`./gradlew test`)를 실행하지 말 것.
- 변경된 기능과 **직접 관련된 테스트 파일만** 선택적으로 실행할 것.

### 5. 작업 완료 후 — 재검토 필수

- 작업 완료 시 반드시 **한 번 더 재검토**를 진행하여 오류를 최소화할 것.
- 체크 항목:
  - [ ] 작업 계획서와 구현 내용이 일치하는가
  - [ ] 불필요한 코드 또는 디버그 로그가 남아있지 않은가
  - [ ] 관련 테스트가 통과했는가
  - [ ] 사이드 이펙트 가능성은 없는가

### 6. 커밋 — 사용자 확인 후 진행

- 코드 작성 완료 후 **커밋을 자동으로 실행하지 말 것**.
- 변경 내용을 요약하여 사용자에게 제시하고, 확인 후 사용자가 직접 커밋할 것.
- 커밋 메시지 컨벤션(예: Conventional Commits)이 있다면 그에 따라 메시지 초안을 제안할 것.

## 모델 사용 규칙

작업 특성에 따라 Opus(계획) / Sonnet(구현)을 분리해 사용한다.

| 단계 | 권장 모델 | 근거 |
| --- | --- | --- |
| 계획서 작성, 아키텍처 설계, 트레이드오프 검토, 코드 리뷰 | `claude-opus-4-7` | 복잡한 의사결정 품질 우선 |
| 계획서 기반 단순 구현, 코드 수정, 테스트 실행 | `claude-sonnet-4-6` | 빠른 실행 + 토큰 비용 효율 |

### 운영 원칙

1. **계획 ↔ 구현 세션은 가급적 분리할 것**
   - 계획 세션: Opus로 시작 → 계획서(`YYMMDD_*.md`)를 `backend/docs/` 또는 `frontend/docs/`에 저장 후 종료.
   - 구현 세션: Sonnet으로 새로 시작 → 저장된 계획서를 `@경로/파일.md` 형태로 참조하며 구현.

2. **현재 세션이 Opus 상태에서 코드 구현 요청이 들어오면**
   - 즉시 구현을 시작하지 말고 아래 안내를 먼저 출력할 것:
     > 현재 모델이 `claude-opus-4-7`입니다. 계획서 기반 구현은 `claude-sonnet-4-6`으로 전환 후 진행을 권장합니다.
     > `/model claude-sonnet-4-6` 실행 후 다시 요청해 주세요.
   - 사용자가 "Opus로 그대로 진행"을 명시하면 구현 가능.

3. **계획서 작성 완료 직후 응답 마지막에 반드시 안내할 것**
   - 다음 문구를 출력해 사용자가 모델 전환 시점을 인지하도록 할 것:
     > 계획서 작성이 완료되었습니다. 구현 단계로 넘어가기 전 `/model claude-sonnet-4-6`으로 전환 후 새 세션에서 계획서를 참조해 구현을 시작하는 것을 권장합니다.

4. **구현 중 아래 상황이 발생하면 Opus 전환을 역으로 제안할 것**
   - 계획서에 없던 설계 변경이 필요해진 경우
   - 여러 구현안 중 트레이드오프 비교가 필요한 경우
   - 의존성/아키텍처 경계가 흔들리는 리팩터링이 필요한 경우
   - 안내 문구:
     > 이 변경은 설계 판단이 필요해 보입니다. `claude-opus-4-7`로 전환 후 계획서를 보강하는 것을 권장합니다.

## 디렉토리 구조 요약

```
myblog-boot/
├── backend/
│   ├── docs/          # backend 작업 계획서 및 문서
│   └── src/
├── frontend/
│   ├── docs/          # frontend 작업 계획서 및 문서
│   └── src/
└── CLAUDE.md
```
