> 작성일: 2026-03-24
> 대상: 기술 면접, 포트폴리오 설명

## 들어가며 — "오버 엔지니어링"을 먼저 인정하라

면접에서 가장 위험한 실수는 과도한 설계를 숨기거나 당연한 것처럼 포장하는 것입니다.
오히려 **먼저 인정하고, 왜 그 선택을 했는지 설명하는 것**이 훨씬 강한 인상을 줍니다.

**추천 오프닝 멘트:**
> "이 프로젝트는 개인 블로그임에도 불구하고 몇 가지 실무 수준의 기술을 의도적으로 적용했습니다.
> 트래픽이 거의 없는 상황에서 Redis나 HMAC 같은 기술은 분명 과설계입니다.
> 하지만 저는 이 프로젝트를 **단순히 동작하는 서비스**가 아니라, **실무 환경을 가정하고 트레이드오프를 직접 경험하는 학습 도구**로 설계했습니다."

이 한 마디로 면접관에게 세 가지를 전달합니다:
1. 트레이드오프를 인식하는 개발자
2. 실무 상황을 고려하는 개발자
3. 학습에 목적의식이 있는 개발자

## 1. Redis — 캐시 도입

### 무엇을 했나
게시글 목록, 게시글 상세, 좋아요 수, 조회수를 Redis에 캐싱.
`board:{id}`, `board:{id}:views`, `board:{id}:likes` 형태의 멀티키 패턴.
Lettuce 클라이언트 + GenericJackson2JsonRedisSerializer 기반 JSON 직렬화.

### 왜 오버 엔지니어링인가
개인 블로그는 동시 요청이 거의 없습니다. DB 인덱스 최적화만으로도 충분합니다.
Redis를 운영하면 오히려 메모리 비용, 캐시 일관성 관리 복잡도가 증가합니다.

### 면접 어필 포인트

**"그럼에도 Redis를 쓴 이유가 있나요?"** 라는 질문에 이렇게 답하세요:

> "실무에서 캐시를 도입할 때 가장 어려운 부분은 기술 자체가 아니라 **Cache Miss 처리, 직렬화 전략, 만료 정책** 같은 운영 이슈입니다.
> 저는 그 경험을 직접 해보고 싶었습니다.
> 예를 들어, 서버 재시작 시 캐시가 비워지는 **Cold Cache 상황에서 DB fallback 처리**, 스케줄러를 통한 **캐시-DB 동기화(Double Write)**, 캐시 패턴 조회 시 Redis SCAN 명령어 사용(KEYS 명령어 블로킹 방지) 같은 실무 문제를 직접 설계하고 해결했습니다."

**핵심 어필 키워드:**
- Cold Cache → DB Fallback
- Double Write 패턴 (Redis → DB 스케줄 동기화)
- SCAN vs KEYS (운영 안전성 고려)
- Lettuce vs Jedis 선택 이유 (Non-blocking I/O)

## 2. HMAC 기반 Stateless 조회수 중복 방지

### 무엇을 했나
게시글 조회수 중복 카운팅을 방지하기 위해 HMAC-SHA256 서명 쿠키를 사용.
쿠키 페이로드: `YYYY-MM-DD:UUID:BASE64_SIGNATURE`
서버에서 서명을 검증하므로 Redis나 DB에 상태를 저장하지 않음.
자정(KST) 기준 일별 자동 만료.

### 왜 오버 엔지니어링인가
개인 블로그에서는 단순히 쿠키에 조회한 게시글 ID 목록을 저장해도 됩니다.
HMAC 검증은 위변조 방지가 목적인데, 개인 블로그 조회수 조작은 보안 위협이 아닙니다.

### 면접 어필 포인트

**"왜 굳이 HMAC을 썼나요?"** 라는 질문에:

> "처음엔 Redis에 방문자 토큰을 저장하는 Stateful 방식으로 구현했습니다.
> 그런데 수평 확장(Scale-out) 환경에서는 Redis 클러스터나 세션 공유 없이는 서버마다 상태가 달라지는 문제가 생깁니다.
> 그래서 서버에 상태를 전혀 두지 않는 **Stateless 방식**으로 전환하면서 HMAC을 선택했습니다.
> 이 과정에서 Stateful과 Stateless의 트레이드오프, 즉 **구현 단순성 vs 확장성**을 직접 비교할 수 있었습니다."

**핵심 어필 키워드:**
- Stateful → Stateless 전환 경험
- 수평 확장(Scale-out)을 고려한 설계
- HMAC-SHA256, 서명 검증
- KST 타임존 처리 (글로벌 서비스 고려)

## 3. 방문자 카운트 — Double Write + 스케줄러 동기화

### 무엇을 했나
방문자 수를 Redis에 먼저 기록(Write-Through)하고,
10분마다 + 자정마다 스케줄러가 Redis 데이터를 DB에 동기화.
Cold Cache 시 DB에서 읽어와 Redis에 적재.
ReentrantLock으로 스케줄러 중복 실행 방지.

### 면접 어필 포인트

> "방문자 카운트는 매 요청마다 DB에 쓰면 쓰기 부하가 생깁니다.
> 실무에서 자주 쓰는 패턴인 **Write Buffer(Redis) → 배치 Flush(DB)** 구조를 직접 구현해봤습니다.
> 이 과정에서 **Eventually Consistent(최종 일관성)** 개념을 체감했고, 서버 장애 시 Redis 데이터 유실 리스크도 함께 고민했습니다."

**핵심 어필 키워드:**
- Write Buffer 패턴
- Eventually Consistent
- ReentrantLock (동시성 제어)
- `@Profile("!test")` (테스트 환경 분리)

## 4. Querydsl — 타입 세이프 동적 쿼리

### 무엇을 했나
`BoardQuerydslRepositoryImpl`, `CategoryQuerydslRepositoryImpl` 등 Querydsl 기반 커스텀 레포지토리.
`fetchJoin` + `distinct()`로 1+N 문제 해결.
`BooleanExpression` 동적 조건 조합.

### 왜 오버 엔지니어링인가
단순 CRUD 블로그에서는 Spring Data JPA의 기본 메서드로 충분합니다.

### 면접 어필 포인트

> "JPQL이나 `@Query` 어노테이션은 조건이 복잡해질수록 문자열 쿼리가 길어지고 컴파일 타임 검증이 안 됩니다.
> Querydsl은 **IDE 자동완성, 컴파일 타임 오류 감지, BooleanExpression 재사용**이 가능해서 유지보수 측면에서 훨씬 안전합니다.
> 특히 카테고리별 게시글 필터링처럼 조건이 동적으로 달라지는 쿼리에서 Querydsl의 장점을 체감했습니다."

**예상 꼬리 질문:** "fetchJoin + distinct가 왜 필요한가요?"

> "`@OneToMany` 컬렉션을 fetchJoin하면 카테고리 1개에 게시글 N개가 있을 때 결과 행이 N배로 뻥튀기됩니다.
> Hibernate 레벨에서 `distinct()`를 적용해 중복 엔티티를 제거했습니다."

## 5. Testcontainers — 실제 Redis 컨테이너로 통합 테스트

### 무엇을 했나
`AbstractContainerBaseTest`에서 `redis:7-alpine` Docker 컨테이너를 싱글턴으로 띄움.
`@DynamicPropertySource`로 host/port를 동적 주입.
18개 테스트 클래스, 99개 테스트 전부 통과.

### 면접 어필 포인트

> "Redis를 Mocking하면 테스트가 빠르지만, Mock과 실제 Redis의 동작 차이(SCAN 커서, TTL 정밀도, 직렬화 포맷)로 인해 프로덕션에서 버그가 나올 수 있습니다.
> Testcontainers로 실제 Redis 컨테이너를 사용해 **실환경과 동일한 조건에서 통합 테스트**를 구성했습니다.
> 싱글턴 컨테이너 패턴을 적용해 전체 테스트 슈트에서 컨테이너를 한 번만 기동, 테스트 속도도 최적화했습니다."

**핵심 어필 키워드:**
- Mock vs 실제 컨테이너 트레이드오프
- Singleton Container 패턴
- `@DynamicPropertySource`

## 6. JWT 인증 — Access Token + Security Filter Chain

### 무엇을 했나
JJWT 0.12.6 기반 HMAC-SHA256 서명.
`JwtFilter`에서 토큰 파싱 → `SecurityContext` 등록.
만료 토큰 전용 커스텀 예외(`InvalidateTokenException`) → 401 응답.

### 면접 어필 포인트

> "세션 방식은 서버에 상태를 저장하므로 수평 확장 시 세션 공유 문제가 생깁니다.
> JWT는 Stateless하므로 어느 서버에서든 토큰만 있으면 인증이 가능합니다.
> 단, JWT는 서버 측 무효화가 어렵다는 단점이 있습니다. 이를 보완하려면 Refresh Token + Redis 블랙리스트가 필요한데, 현재는 구현하지 않았고 그 이유는 개인 블로그 규모에서 복잡도 대비 효용이 낮다고 판단했기 때문입니다."

이 답변이 강력한 이유: **"구현 안 한 이유를 트레이드오프로 설명"** — 단순히 몰라서가 아님을 보여줌.

## 7. TanStack Query + Redux Toolkit — 프론트엔드 상태 관리 이원화

### 무엇을 했나
- 서버 상태: TanStack Query v5 (게시글, 카테고리, 댓글)
- 클라이언트 상태: Redux Toolkit v2 + redux-persist (로그인 사용자 정보)
- queryKeys Factory 패턴으로 계층적 캐시 무효화
- Axios 인터셉터로 토큰 자동 주입

### 면접 어필 포인트

> "React 앱에서 상태를 하나의 도구로 관리하면 서버 데이터와 UI 상태가 혼재해 복잡도가 높아집니다.
> 서버 상태(비동기 데이터)와 클라이언트 상태(UI 상태)를 분리하는 것이 현재 프론트엔드 아키텍처의 트렌드입니다.
> TanStack Query로 서버 데이터의 캐싱, 동기화, 로딩/에러 상태를 처리하고, Redux는 로그인 정보처럼 앱 전역에서 필요한 클라이언트 상태만 담당하도록 역할을 분리했습니다."

### "이 기술이 진짜 필요했나요?" 에 대한 답변 구조

```
1. 인정: "개인 블로그 규모에서는 ___는 과설계입니다."
2. 의도: "하지만 ___ 상황을 가정하고 직접 경험해보고 싶었습니다."
3. 학습: "이 과정에서 ___ 문제를 발견하고 ___ 방식으로 해결했습니다."
4. 트레이드오프: "다만 ___라는 단점이 있고, 실제 도입 시에는 ___ 조건에서만 권장합니다."
```

### 가장 강한 어필 포인트 TOP 3

| 순위 | 포인트 | 이유 |
|------|--------|------|
| 1 | Stateful → Stateless HMAC 전환 | 확장성 고민 + 실제 리팩토링 경험 |
| 2 | Testcontainers 통합 테스트 | Mock 한계를 인식하고 실환경 테스트 구축 |
| 3 | Double Write + 스케줄러 동기화 | DB/캐시 일관성 문제를 스스로 설계 |

### 피해야 할 답변 패턴

- ❌ "Redis는 성능이 좋으니까 썼습니다" (이유가 빈약)
- ❌ "스펙에 넣으면 좋아 보여서요" (정직하지 못함)
- ❌ "사실 잘 모르지만..." (자신감 저하)
- ✅ "개인 블로그에는 과설계지만, ___ 상황을 가정하고 ___ 문제를 직접 해결해보기 위해 선택했습니다"

## 운영 관점에서 어필할 수 있는 요소

이 프로젝트는 포트폴리오인 동시에 **실제 운영 중인 서비스**입니다.

- **실제 트래픽 처리**: 운영 중에 발생한 버그(Cold Cache, 카테고리 중복 표시 등)를 직접 발견하고 수정
- **점진적 개선**: Redis Stateful → HMAC Stateless 전환처럼 더 나은 방향으로 계속 리팩토링
- **테스트 안전망**: 99개 테스트가 실제 컨테이너 기반으로 통과 — 리팩토링 시 회귀 방지

> "이 프로젝트는 동작만 하는 프로젝트가 아닙니다.
> 실제로 운영하면서 발견한 버그를 수정하고, 더 나은 설계를 계속 탐구하는 **살아있는 프로젝트**입니다."
