# CLAUDE.md

## 작업 범위 지정 커맨드

메시지 맨 앞에 아래 prefix를 붙여 작업 범위를 지정한다.

| prefix | 의미          | 문서 경로        | 작업 디렉토리 |
| ------ | ------------- | ---------------- | ------------- |
| `-b`   | backend 작업  | `backend/docs/`  | `backend/`    |
| `-f`   | frontend 작업 | `frontend/docs/` | `frontend/`   |

- prefix가 없으면 반드시 작업 범위를 사용자에게 되물을 것.
- 두 영역에 걸치는 작업이면 `-b -f` 또는 `-bf` 로 표기한다.

---

## 워크플로우 규칙

### 1. 문서 위치 규칙

- `backend` 관련 문서는 반드시 `backend/docs/` 폴더에 작성할 것.
- `frontend` 관련 문서는 반드시 `frontend/docs/` 폴더에 작성할 것.
- 문서 파일명에는 항상 `YYMMDD_` 형식의 날짜 prefix를 붙일 것. (시스템의 현재 날짜를 확인할 것)
  - 예: `260323_board-view-count-cookie-plan.md`

### 2. 코드 작성 이전 — 작업 계획서 필수

- 코드를 작성하기 전, 반드시 **작업 계획서**를 markdown 형식으로 상세히 작성할 것.
- 계획서에는 다음 내용을 포함할 것:
  - 작업 목표 및 배경
  - 구현 접근 방식 및 설계 결정
  - 변경 대상 파일 목록
  - 예상되는 이슈 및 대응 방안

### 3. 코드 작성 시 — Context7 MCP 활용

- 코드 작성 시 **Context7 MCP**를 적극 활용하여 최신 라이브러리 문서를 참조할 것.
- 공식 문서 기반으로 작성하여 버전 불일치 및 오류를 사전에 방지할 것.

### 4. 테스트 — 관련 파일만 실행

- 코드 작성 완료 후 전체 테스트(`./gradlew test`)를 실행하지 말 것.
- 변경된 기능과 **직접 관련된 테스트 파일만** 선택적으로 실행할 것.

### 5. 작업 완료 후 — 재검토 필수

- 작업 완료 시 반드시 **한 번 더 재검토**를 진행하여 오류를 최소화할 것.
- 체크 항목:
  - [ ] 작업 계획서와 구현 내용이 일치하는가
  - [ ] 불필요한 코드 또는 디버그 로그가 남아있지 않은가
  - [ ] 관련 테스트가 통과했는가
  - [ ] 사이드 이펙트 가능성은 없는가

### 6. 커밋 — 사용자 확인 후 진행

- 코드 작성 완료 후 **커밋을 자동으로 실행하지 말 것**.
- 변경 내용을 요약하여 사용자에게 제시하고, 확인 후 사용자가 직접 커밋할 것.
- 커밋 메시지 컨벤션(예: Conventional Commits)이 있다면 그에 따라 메시지 초안을 제안할 것.

## 디렉토리 구조 요약

```
myblog-boot/
├── backend/
│   ├── docs/          # backend 작업 계획서 및 문서
│   └── src/
├── frontend/
│   ├── docs/          # frontend 작업 계획서 및 문서
│   └── src/
└── CLAUDE.md
```
