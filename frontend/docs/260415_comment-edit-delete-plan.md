# 댓글 수정/삭제 UI 구현 계획서

## Problem (문제 정의)

### 어떤 문제를 해결하려는가?
- 백엔드 API(PUT/DELETE `/api/v1/comments/{commentId}`)와 프론트엔드 API 함수(`editComment`, `deleteComment`)는 이미 구현되어 있으나, **사용자가 댓글을 수정/삭제할 수 있는 UI가 없다.**
- 댓글 작성 시 "비밀번호 (수정/삭제에 사용)"이라고 안내하지만 실제로 사용할 수 없는 상태.

### 해결하지 않을 경우?
- 게스트 사용자는 잘못된 댓글을 수정/삭제할 방법이 없음
- 관리자도 부적절한 댓글을 프론트엔드에서 직접 삭제할 수 없음

---

## Analyze (분석 및 선택지 검토)

### 현재 상태

| 계층 | 상태 |
|------|------|
| 백엔드 Controller/Service | 완료 (PUT/DELETE 엔드포인트) |
| 프론트엔드 API 함수 | 완료 (`editComment`, `deleteComment` in `postApi.ts`) |
| 프론트엔드 타입 | **미완료** — `Comment` 타입에 `isAdmin`, `modificationStatus` 필드 누락 |
| 프론트엔드 UI | **미구현** |

### 인증 방식 (백엔드 기준)

| 사용자 | 수정/삭제 인증 |
|--------|---------------|
| 관리자 (JWT 보유) | 비밀번호 불필요 — `Principal != null`이면 바이패스 |
| 게스트 | 댓글 작성 시 입력한 4자리 비밀번호 필요 |

### UI 접근 방식

**선택지 A: 모달(다이얼로그) 방식으로 비밀번호 입력**
- 장점: 기존 댓글 레이아웃을 건드리지 않음, 깔끔한 UX
- 단점: 모달 컴포넌트 추가 필요

**선택지 B: 인라인으로 비밀번호 입력 필드 표시**
- 장점: 모달 없이 간단, 댓글 바로 아래에 표시
- 단점: 레이아웃이 다소 복잡해질 수 있음

**선택: B (인라인 방식)**
- 이유: 기존 reply-form 패턴과 일관성 유지, 별도 모달 컴포넌트/오버레이 불필요, 구현 단순

### 수정 UI 접근 방식

**인라인 편집**: 댓글 텍스트 영역을 textarea로 교체하여 바로 편집
- 기존 CommentForm과 별개로, 수정 전용 인라인 UI 사용
- 수정 시 `(수정됨)` 표시 → 백엔드 `modificationStatus: "MODIFIED"` 활용

## Action (구현 계획)

### 작업 범위
1. `types/index.ts` — `Comment` 인터페이스에 `isAdmin`, `modificationStatus` 추가
2. `Comment.tsx` — 수정/삭제 버튼 + 인라인 수정 폼 + 비밀번호 입력 + 삭제 확인 UI
3. `globals.css` — 수정/삭제 관련 스타일 추가
4. 자식 댓글(reply-item)에도 동일한 수정/삭제 기능 적용

### 변경 대상 파일
| 파일 | 변경 내용 |
|------|----------|
| `frontend/types/index.ts` | `Comment` 타입에 `isAdmin`, `modificationStatus` 필드 추가 |
| `frontend/components/comments/Comment.tsx` | 수정/삭제 UI 로직 추가 |
| `frontend/app/globals.css` | 수정/삭제 관련 CSS 추가 |

### 상세 동작 흐름

#### 수정 흐름
1. 댓글 우측에 `수정` | `삭제` 버튼 표시
2. `수정` 클릭 → 게스트: 비밀번호 입력 필드 표시 → 비밀번호 입력 후 편집 모드 진입 (또는 바로 편집 모드 + 저장 시 비밀번호)
   - 더 나은 UX: 바로 편집 모드 진입, 저장 시 비밀번호 함께 전송
3. 관리자: 바로 편집 모드 진입 (비밀번호 불필요)
4. 편집 모드: 댓글 내용이 textarea로 변경 + 하단에 게스트용 비밀번호 필드 + 저장/취소 버튼
5. 저장 성공 시 → 캐시 무효화, 편집 모드 해제
6. 실패 시 → 에러 메시지 표시 (비밀번호 불일치 등)

#### 삭제 흐름
1. `삭제` 클릭 → 확인 UI 표시 (게스트: 비밀번호 입력 + 삭제 확인 버튼, 관리자: 삭제 확인 버튼만)
2. 확인 후 API 호출 → 성공 시 캐시 무효화
3. 실패 시 → 에러 메시지 표시

### 트레이드오프
- 인라인 방식으로 모달 없이 구현 → 간단하지만 댓글 영역이 일시적으로 확장됨
- 자식 댓글도 동일 컴포넌트로 처리 → 코드 재사용성 확보

## Result (검증 계획)

### 테스트 시나리오
1. **관리자 수정**: 로그인 상태에서 댓글 수정 → 비밀번호 없이 성공
2. **관리자 삭제**: 로그인 상태에서 댓글 삭제 → 비밀번호 없이 성공
3. **게스트 수정**: 비로그인 상태에서 댓글 수정 → 올바른 비밀번호로 성공 / 틀린 비밀번호로 실패
4. **게스트 삭제**: 비로그인 상태에서 댓글 삭제 → 올바른 비밀번호로 성공 / 틀린 비밀번호로 실패
5. **자식 댓글**: 답글에 대해서도 수정/삭제 동작 확인
6. **수정됨 표시**: 수정된 댓글에 `(수정됨)` 표시 확인
7. **캐시 무효화**: 수정/삭제 후 목록이 정상 갱신되는지 확인

### 성공 기준
- 관리자/게스트 모두 수정/삭제 가능
- 비밀번호 불일치 시 적절한 에러 메시지
- 수정 후 `(수정됨)` 표시
- 기존 댓글 작성/답글 기능에 영향 없음
